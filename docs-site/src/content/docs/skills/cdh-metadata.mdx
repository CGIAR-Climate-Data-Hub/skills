You point the agent at a geospatial file. It inspects it automatically, asks you the fields it
cannot derive, shows you a plan, and writes a `.yaml` file that validates against the
[CDH metadata schema](https://cgiar-climate-data-hub.github.io/cdh-metadata-standard/v0.1.0/schemas/core.schema.json).

The skill handles **GeoTIFF, NetCDF, Zarr, GeoPackage, Shapefile, FlatGeobuf, and Parquet** inputs.

## What auto-extracts vs. what you answer

| Source | Fields filled automatically |
|--------|-----------------------------|
| File inspection | Bounding box, CRS, spatial resolution, variable names, data types, nodata value, file size, media type |
| Always asked | `id`, `title`, `description`, `license`, contact (licensor), citation or DOI, data URLs, CDH domain |
| Asked only if missing | Temporal range, temporal resolution, keywords, version, processing provenance, geography |

## Four-stage workflow

**Stage 1 — Inspect.** The agent runs a short Python snippet (rasterio, xarray, or geopandas
depending on file type) and prints the extracted metadata for you to verify.

**Stage 2 — Collect inputs.** Two rounds of questions are asked together, not one at a time.
Round 1 covers the always-required fields. Round 2 asks only for anything still unknown.

**Stage 3 — Confirm.** A compact plan is shown before any file is written:

```
ID:       my-dataset-v1
Title:    My Dataset
License:  CC-BY-4.0
Temporal: 2020-01-01 → 2022-12-31
Spatial:  [-90.0, 10.0, -75.0, 18.0] — EPSG:4326
Domain:   climate
Data:     https://example.org/data/my-dataset.tif
Output:   D:/data/my-dataset-v1.yaml
```

**Stage 4 — Generate.** The YAML file is written to the same directory as the dataset (or
wherever you specify), named `<id>.yaml`.

## CDH domains

| Value | Description |
|-------|-------------|
| `adaptation` | Climate adaptation measures |
| `agricultural-production` | Crop and livestock production |
| `boundaries` | Administrative or geographic boundaries |
| `climate` | Climate variables and projections |
| `hydrology` | Water resources |
| `mitigation` | GHG mitigation |
| `socioeconomic` | Socioeconomic indicators |

## Supported extensions

The skill automatically adds the correct extension URLs to `extensions[]` in the YAML based on
which blocks are used:

| Extension | When it's added |
|-----------|----------------|
| `cdh` | Always — when a `cdh:` block is present |
| `datacube` | When `dimensions:` or `variables:` blocks are used (NetCDF, Zarr) |
| `agriculture` | When `commodities:` is specified |
| `climate` | When a `climate:` block is included |
| `classification` | When a `classes:` field is present |

<TabItem label="Claude Code">

Complete the [Claude Code setup](../usage/claude-code/) first, then launch `claude` from the project root.

**Example request — GeoTIFF**

```
Create CDH metadata for D:/data/MapaForestal_2024.tif.
This is Honduras National Forest Map 2024, published by ICF under CC-BY-4.0.
Contact: Juan García, ICF, jgarcia@icf.gob.hn
DOI: 10.12345/forestal-2024
The data is at https://s3.amazonaws.com/cdh-data/MapaForestal_2024.tif
Domain: boundaries
```

The skill inspects the file, asks any missing fields, shows a plan, and writes
`MapaForestal_2024.yaml` next to the source file.

**Example request — NetCDF**

```
Generate CDH metadata for D:/data/yield_dedza_bean.nc
This is DSSAT bean yield simulation output for Dedza district, Malawi (2000–2020).
License: CC0-1.0, produced by CIMMYT. Domain: agricultural-production
```

For a NetCDF the skill additionally fills in the `datacube` extension with dimension
and variable metadata extracted from the file.

**Trigger phrases**

The skill also activates on informal requests:

```
document this raster for CDH
write a YAML for my soil file
how do I add my dataset to the climate data hub
help me fill in the metadata fields
```

:::tip[No aggeodata needed]
This skill uses standard Python libraries (`rasterio`, `xarray`, `geopandas`) and writes plain
YAML — it does not require `aggeodata` to be installed.
:::
